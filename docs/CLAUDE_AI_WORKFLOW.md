# win-launcher-site 開発ワークフロー(Claude.ai向け)

このファイルは、Claude.ai(chat)がwin-launcher-siteの要件変更を受け付ける際に
「まず読む」ファイルです。Claude.aiの記憶(userMemories)にはこのファイルへの
参照ルールのみを残し、詳細はここに集約します。

サイトのコード実装上の技術詳細・落とし穴は、このファイルではなく
リポジトリ直下の `CLAUDE.md`(Claude Code=CC向け)が正本です。
このファイルは「誰が何をするか」「どういう手順で進めるか」という
プロセス・運用ルールに専念します。

## 役割分担

- **Claude.ai(このファイルを読む側)**: 要件・レイアウトの設計議論、
  (必要なら)ビジュアルモックアップの提示、CCへの実装プロンプート作成、
  push指示プロンプートの作成。
- **CC(Claude Code CLI)**: 実際のコード変更、`npm run build` によるビルド確認、
  レイアウト・見た目に関わる変更の場合はPlaywrightでの目視確認、
  git commit、(指示があれば)git push、リポジトリ側CLAUDE.mdへの知見追記。

## 基本ワークフロー

1. ユーザーがサイトへの要件変更・追加を発言
2. Claude.aiが現状のサイト(本番URLまたはリポジトリ)を確認し、必要なら
   デザイン・レイアウトの方向性を(モックアップ等で)提案、ユーザーと合意
3. Claude.aiがCC向けの実装プロンプートを作成
   - レイアウト・見た目に関わる変更の場合、プロンプートには必ず
     「実装後、Playwrightでデスクトップ幅・モバイル幅それぞれの
     スクリーンショットを取得し目視確認する」ことを含める
   - プロンプート末尾は `npm run build` を通すことの確認で締める
   - 変更内容に応じてCC側CLAUDE.mdへの知見追記も1つの指示にまとめて含める
4. ユーザーがCCにプロンプートを渡し、実装・ビルド確認・報告を受け取る
5. ユーザーが結果を確認(スクリーンショット、必要ならローカル確認)
6. 問題なければ、Claude.aiが **git commit + git push** の指示プロンプートを作成
   (実装プロンプートに含めてもよいし、別出しでもよい。ただし
   「pushしないと本番に反映されない」ことを毎回明示する)
7. push後、GitHub Actionsのビルドが数分で完了し、本番URLに反映される
